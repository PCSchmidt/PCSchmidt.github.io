## Working Agreements

- Keep all portfolio claims evidence-backed.
- Use `featured` only for projects with runnable setup and proof artifacts.
- Mark incomplete but promising work as `in-progress`.
- Prefer small daily merges over large end-of-week pushes.

## Definition of Done (Week 1)

- Tiering is finalized (Tier 1, Tier 2, Tier 3).
- Each selected repo has a completion issue list.
- At least one repo has:
  - improved README
  - confirmed run instructions
  - architecture diagram draft
  - sample outputs committed
- Portfolio copy is updated to match actual repo state.

## Risk Register (Week 1)

1. Risk: Overcommitting to too many repos.
   - Mitigation: Focus on one flagship completion first.

2. Risk: Claims exceed implemented functionality.
   - Mitigation: Tie every claim to artifact (README, screenshot, output, metric).

3. Risk: Setup friction for reviewers.
   - Mitigation: Add strict quickstart steps and tested prerequisites.

4. Risk: Incomplete projects looking unfinished on portfolio.
   - Mitigation: Use `in-progress` badge + clear next iteration section.

## End-of-Week Review Questions

1. Can a recruiter open two links and see clear technical depth?
2. Can an engineer run at least one flagship project without confusion?
3. Does the portfolio accurately reflect current repo maturity?
4. Is the next week plan obvious from open repo issues?
